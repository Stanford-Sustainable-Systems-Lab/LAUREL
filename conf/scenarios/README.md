# conf/scenarios/

> **Most subdirectories here are generated — but some can be hand-built (see below).**

Subdirectories here contain per-task parameter overrides used when running individual scenario tasks. Most are created by the `build_scenarios` Kedro pipeline:

```
kedro run --pipeline=build_scenarios --env=build_scenarios/<name>
```

Each subdirectory maps to one SLURM array task:

```
conf/scenarios/<scenario_name>/task_<N>/parameters.yml
```

The corresponding hand-written build specifications (which builder class to use, what parameter ranges to sweep) live in [`conf/build_scenarios/`](../build_scenarios/).

To regenerate configs for a scenario family, run:

```bash
kedro run --pipeline=build_scenarios --env=build_scenarios/<name>
```

The matching SLURM batch script is written to `scripts/<name>.sh`.

## Hand-built scenario families

Not all scenario families need to be generated by the `build_scenarios` pipeline. You can hand-build a scenario (or family of scenarios) by creating the subdirectory structure manually. The only requirement is that the bottom-level folders follow the `task_<N>` naming convention so that the scenario runner can discover them:

```text
conf/scenarios/<scenario_name>/task_0/parameters.yml
conf/scenarios/<scenario_name>/task_1/parameters.yml
...
```

Hand-built families should be committed to version control (unlike generated ones, which are gitignored). To track a hand-built family, add the following to `.gitignore`:

```gitignore
!conf/scenarios/<scenario_name>/
!conf/scenarios/<scenario_name>/**
```
